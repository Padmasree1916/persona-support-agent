Designing a Reliable Retrieval-Augmented Generation System

Introduction

Retrieval-Augmented Generation, commonly called RAG, combines information retrieval with language-model generation. A normal language model answers from patterns learned during training, which makes it useful for general questions but unreliable for private, recent, or highly specialized information. A RAG system changes the workflow. Before asking the model to answer, it searches a controlled document collection and supplies the most relevant passages as context. The model is then instructed to use those passages and cite their origins. This design does not make a model infallible, but it creates a traceable path from a question to supporting evidence.

Document ingestion

The first stage is ingestion. Source files may be PDF reports, Word documents, text files, policies, manuals, or research notes. Each format requires a suitable parser. PDF processing should preserve page numbers because users often need to verify a quotation or claim in the original file. Word documents do not always have stable page boundaries, so paragraph or section numbers are often more reliable metadata. Plain-text files can be represented as one or more logical sections. Extraction should normalize repeated whitespace while retaining paragraph boundaries. Scanned PDFs require optical character recognition because they contain images rather than an accessible text layer.

Metadata is as important as extracted text. Every unit should carry the filename and a location such as a page, heading, paragraph, or section. Optional fields can include title, author, publication date, department, access level, and document version. Metadata enables citations and can also constrain retrieval. For example, an application may search only policies issued by a particular department or documents published after a given date.

Chunking strategy

Embedding an entire document as one vector usually produces poor retrieval because a long file contains many unrelated ideas. The text should instead be divided into chunks. A useful recursive strategy first attempts to split at paragraph boundaries, then line breaks, sentence boundaries, spaces, and finally individual characters. This keeps coherent ideas together more often than blindly cutting every fixed number of characters.

Chunk size involves a tradeoff. Very small chunks are precise but may omit definitions or qualifications needed to interpret a sentence. Very large chunks preserve context but introduce noise and consume more of the language model's context window. Overlap protects information near boundaries by repeating a small amount of text in adjacent chunks. A practical initial setting is approximately 700 to 1,200 characters with 100 to 200 characters of overlap. The best values should be selected through evaluation rather than intuition alone.

Embeddings and storage

An embedding converts text into a numeric vector. Similar passages should produce vectors that are close according to a distance measure such as cosine distance. The same embedding process must be used for document chunks and user questions. If indexing and querying use different vector spaces, similarity scores become meaningless. Documents should be embedded in batches to reduce overhead and improve throughput.

A vector database stores vectors, original chunk text, identifiers, and metadata. Persistent storage is important because rebuilding an index can be expensive. Indexing and querying should therefore be separate operations. The indexing command processes documents only when the collection changes, while the query application opens the existing database. Stable chunk identifiers help avoid duplicates. When documents are removed or substantially edited, a controlled rebuild is often safer than leaving stale chunks in the collection.

Retrieval and prompting

At query time, the application embeds the question and retrieves the top matching chunks. The number of results, usually called top-k, should be configurable. A small value can miss supporting evidence, while a large value may fill the prompt with irrelevant passages. Similarity thresholds can reject obviously unrelated results, although thresholds need calibration for the chosen model and dataset. More advanced systems may combine vector similarity with keyword search, apply metadata filters, or rerank an initial candidate set.

The prompt should clearly separate instructions, context, and the user's question. Each context block should display its source metadata. The model must be told to use only the supplied context, cite factual statements, and give a consistent not-found response when evidence is absent. The application should show both the final answer and retrieved chunks. This transparency lets a user distinguish a retrieval error from a generation error.

Operations and security

A production RAG system needs monitoring and access control. Logs can record latency, retrieved identifiers, model errors, and user feedback without storing sensitive question content unnecessarily. Permissions must be applied before retrieval; hiding a citation after generation is too late if restricted text has already entered the prompt. Secrets belong in environment variables or a managed secret store, never source control. Teams should also define retention policies for indexes and logs.

The most dependable systems favor understandable components, reproducible indexing, and visible evidence. Good retrieval cannot compensate for poor source documents, and fluent generation cannot compensate for irrelevant retrieval. Reliability comes from treating ingestion, metadata, chunking, embedding, retrieval, prompting, and evaluation as connected engineering decisions.
